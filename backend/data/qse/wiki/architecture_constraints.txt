1. STREAMING & DELIVERY ARCHITECTURE
=============================================================

1.1 CDN Strategy
- Primary CDN: Akamai (global). Failover CDN: Fastly.
- DNS-based routing currently in use for CDN switching (5–10 min propagation).
- Content is packaged in both HLS (Apple clients) and DASH (Android, Web, Smart TV).
- Target: Migrate to CMAF (Common Media Application Format) to serve a single fragmented MP4 stream to all client types. CMAF reduces storage footprint by ~45%.
- Live stream latency target: <5 seconds via CMAF chunked transfer (current: 30–45s with HLS).

1.2 Transcoding Pipeline
- Encoding stack: AWS Elemental MediaConvert for VOD, AWS Elemental MediaLive for live.
- Output profiles: 240p, 480p, 720p, 1080p, 2160p (4K HDR10 and Dolby Vision where available).
- All content must be packaged with AES-128 encryption as baseline.
- Premium content (4K HDR) must use hardware-backed DRM (Widevine L1 / FairPlay).

1.3 Storage
- Origin storage: AWS S3 (us-east-1 primary, eu-west-1 secondary).
- Total stored content: ~12 petabytes.
- Metadata database: PostgreSQL 15 (Amazon RDS, Multi-AZ).

2. DRM & CONTENT SECURITY ARCHITECTURE
=============================================================

2.1 Current DRM Setup
- Widevine L3 (software DRM) for Android and web.
- FairPlay Streaming for iOS and macOS.
- PlayReady for Windows and Xbox.
- DRM key server: Internal Node.js service using castLabs DRMtoday module.
- Maximum resolution under L3: 1080p.

2.2 L1 DRM Requirements (Planned)
- Widevine L1 requires hardware TEE (Trusted Execution Environment) on device.
- Device attestation: Google Play Integrity API (Android), Apple DeviceCheck (iOS).
- Key server must support Google License Proxy Auth (LPA) protocol.
- FairPlay L1 requires Apple Key Server Module (KSM) certification (4–6 weeks).
- L1 unlocks: 4K HDR10, Dolby Vision, Dolby Atmos.

2.3 Key Server Architecture Constraints
- Key server MUST run in active-active configuration across ≥2 AWS regions before L1 launch.
- All license requests must be audit-logged (ISO 27001 requirement).
- License request anomaly detection required (detect credential sharing / piracy).
- Short-lived license tokens: max 72 hours for premium tier, 24 hours for standard.

2.4 Forensic Watermarking
- Provider: Nagra NexGuard.
- Current approach: Post-CDN origin injection (server-side per-request watermark).
- With L1 DRM, decryption moves to device TEE – watermark must be embedded pre-delivery.
- Architecture decision pending: per-subscriber origin watermarking vs AB-watermarking.
  - Per-subscriber: Higher CDN cache miss rate for premium content.
  - AB-watermarking: Extra storage (~2x premium content) but preserves CDN cache efficiency.
- CONSTRAINT: No change to watermark architecture without Security Architect sign-off (Tariq Nasser).

3. DATA PLATFORM & ML ARCHITECTURE
=============================================================

3.1 Event Streaming
- User events flow: Client → AWS API Gateway → Kinesis Data Streams → Kafka (MSK) → Feature Store.
- Current batch processing: Spark on EMR, 4-hour cadence.
- Target: Streaming features (<5 min latency) using Apache Flink on MSK.
- Feature store: Feast (offline) + Redis (online serving).

3.2 Recommendation Engine
- Current: Collaborative filtering (AWS Personalize).
- Target: Hybrid model = content-based features (metadata, mood, theme) + collaborative signals.
- Cold-start strategy: Use content metadata features until ≥50 user interactions available.
- Content metadata enrichment: AI pipeline (Claude API) over synopsis + reviews → genre, mood, theme tags.
- Metadata confidence threshold: ≥85% precision required before using tags in recommendations.

3.3 Data Governance
- User viewing data is CLASSIFIED as PII under company data policy.
- CONSTRAINT: Viewing data MUST NOT be passed to third-party ad networks without explicit user consent (GDPR, CCPA compliance).
- Contextual ad signals (genre, time of day, geography from IP) are permitted without consent.
- Data retention: Viewing history retained for 24 months per user, then anonymised.

4. AUTH & IDENTITY ARCHITECTURE
=============================================================

4.1 Authentication
- OAuth 2.0 / OIDC with internal IdP (Keycloak, v22).
- Access tokens: 24-hour lifetime (MUST be reduced to <1 hour for service-to-service).
- Refresh tokens: 30 days.
- MFA: Optional for consumer, REQUIRED for internal admin users.

4.2 Service-to-Service Auth
- CRITICAL CONSTRAINT: Long-lived shared tokens are PROHIBITED for service accounts (security policy SEC-04).
- Required: Rotate to short-lived machine tokens with automated refresh. HashiCorp Vault preferred.
- Target token lifetime for service accounts: 15 minutes.

4.3 Profile System
- Up to 5 profiles per account.
- Profile types: Standard, Kids (restricted), Admin (can manage profiles).
- Kids profiles MUST be PIN-protected for switching – CURRENTLY NOT IMPLEMENTED.
- Screen time limits: Not yet implemented.

=============================================================
5. API GATEWAY & PLATFORM STABILITY
=============================================================

5.1 Rate Limiting
- No rate limiting currently implemented on partner API.
- Required: Per-consumer-key rate limits with graduated throttling.
- Circuit breakers required before Q3 partner API expansion.
- SLA target: 99.9% uptime for streaming API.

5.2 Observability
- Log aggregation: OpenSearch (Kibana) – 5 separate dashboards.
- No distributed tracing currently.
- Target: OpenTelemetry instrumentation across all microservices, single trace view in Grafana Tempo.

5.3 API Versioning
- External API: Versioned (/v1/, /v2/).
- CONSTRAINT: Breaking changes to v1 API require 90-day deprecation notice to partners.

6. AD TECH ARCHITECTURE (PLANNED – Q3)
=============================================================

- Ad tiers: Free (ad-supported), Standard, Premium (ad-free).
- SSAI (Server-Side Ad Insertion) vendor: TBD – evaluating AWS Elemental MediaTailor and Yospace.
- Ad decision service: OpenX (primary ad network), Index Exchange (secondary).
- Signal policy: Contextual signals only (genre, content type, geography, time of day).
- CONSTRAINT: No user-level identifiers may be passed to ad decisioning systems.
- CONSTRAINT: SSAI implementation must not break existing HLS/DASH playback compatibility.
- IMA SDK (client-side) deprecated for mid-roll inventory – SSAI required.
